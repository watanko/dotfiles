## 1. 一般事項
- ユーザーは日本語話者です．思考は英語で良いですが，最終的な回答は日本語で行ってください．
- ただし、ファイル内のコメントは英語で端的に書いてください

## 2. 環境とツール類 (Python)
- バージョン管理はuvで行っています．pipを使用しないでください．
- 依存関係は`pyproject.toml`で管理しています．
- 依存を追加する場合は`uv add <パッケージ名>`で追加し，絶対に`pyproject.toml`を直接編集しないでください．
- 開発用の依存は`uv add --dev <パッケージ名>`で追加してください．
- フォーマットにはruffを使用し，`uv run ruff check . --fix && uv run ruff format .`でコードを整形してください．
- 型チェックにはmypyを使用し，`uv run mypy .`で型チェックを行ってください．
- テストにはpytestを使用し，`uv run pytest .`でテストを実行してください．
    - テストカバレッジは，スキーマやViewを除き，90%を目指してください．
    - テスト実装を依頼された際には，カバレッジの達成のために元のコードを変更することはしないでください．
- `pyproject.toml`にはすでにruffの設定が含まれているので，`uv run ruff check . --fix && uv run ruff format .`でコードを整形し，発生した警告を修正してください．
- loggingにはstructlogを使用してください．

## 3. コーディング規約
- 型アノテーションを必ず追加してください．
- 1ファイルが1000行を超えないようにしてください．必要に応じてファイルを分割してください．
- 1ディレクトリにファイルが7つ以上ある場合は、サブディレクトリを作成してファイルを整理してください．
- docstringを必ず追加してください．特にクラス・関数の説明と，Args, Attributes, Returns, Raisesセクションを充実させてください．
- エラーは起こしてください．例外処理でNoneや空リストを返すのは避けてください．また，try catchで再スローも避けてください．try文は非同期処理に関する場合、すなわち一件が落ちると全滅する場合以外は使用禁止です.
- やたらヘルパー関数を多用するのは可読性が下がります．型変換など，多用する場合を除き避けてください．
- コメントを書く場合には何をしているかではなく，why, why notを中心に書いてください
- テストを書く際のコメントは，A should, A should not, A must, A must notのように書いてください
- 修正指示に対してアドホックな実装をするのではなく、今後の拡張可能性を考えた上でRepository・Factory・Strategyパターンを有効活用してください．とにかく振る舞いを記述するクラスと、実装を記述するクラスを分けてください．
- ディレクトリ構成は、特に指示がなければfeature baseな構成にしてください．人間がパッと見でどんな機能があるのか分かるようにしてください．
  - とはいえ、銀の弾丸はないので、プロジェクトの規模や性質に応じて柔軟に対応してください．
